# 🤝 기여 가이드라인 (Contributing Guidelines)

AWS Cleaner 프로젝트에 관심을 가져주셔서 감사합니다! 여러분의 기여는 프로젝트를 발전시키는 데 큰 도움이 됩니다. 아래 절차를 따라 기여해 주시면 감사하겠습니다.

---

## 🐛 버그 신고 및 기능 제안

버그를 발견하셨거나 새로운 기능을 제안하고 싶으신 경우, 직접 코드를 수정하기 전에 **[GitHub Issues](https://github.com/hslcrb/missile_to_AWS/issues)**를 활용해 주세요.

### 1. 버그 신고 (Bug Reports)
이슈를 생성할 때 아래 내용을 포함해 주세요:
- **환경**: Windows 버전, 빌드 환경 (Visual Studio 버전 등)
- **증상**: 기대했던 동작과 실제로 발생한 동작의 차이
- **재현 방법**: 어떤 순서로 조작했을 때 버그가 발생하는지 단계별로 설명
- **에러 메시지/스크린샷**: 가능한 경우 첨부 요망

### 2. 기능 제안 (Feature Requests)
새로운 아이디어가 있다면 아래 내용을 포함해 주세요:
- **제안 배경**: 왜 이 기능이 필요한지?
- **상세 설명**: 기능이 어떻게 동작해야 하는지?
- **예상 이점**: 이 기능이 추가되면 어떤 점이 좋아지는지?

## 🛠️ 개발 및 풀 리퀘스트 (Pull Requests)

직접 코드를 수정하여 프로젝트에 기여하고 싶으신 경우 아래의 워크플로우를 따릅니다.

### 1. 포크 및 브랜치 생성
1. 프로젝트를 **Fork** 합니다.
2. `next-gamma` 브랜치를 기준으로 새로운 브랜치를 생성합니다.
   - 예: `git checkout -b feature/amazing-feature` 또는 `git checkout -b fix/bug-fix`

### 2. 코딩 규칙
- **Pure Win32 & C++**: 가급적 외부 라이브러리 의존성을 추가하지 마세요. 모든 UI는 Win32 API 및 GDI+로 구현하는 것을 원칙으로 합니다.
- **주석 및 명명 규칙**: 기존 코드 스타일(`camelCase` 또는 `PascalCase`)을 따르며, 복잡한 로직에는 설명을 추가해 주세요.
- **보안**: 자격 증명(Credentials)을 평문으로 처리하거나 디스크에 남기는 코드는 지양해 주세요.

### 3. 커밋 메시지 컨벤션
다음과 같은 접두어를 사용하여 커밋의 의도를 명확히 합니다:
- `[FEAT]`: 새로운 기능 추가
- `[FIX]`: 버그 수정
- `[DOCS]`: 문서 수정
- `[UI]`: 사용자 인터페이스 개선
- `[REFACTOR]`: 코드 리팩토링

### 4. PR 제출 전 확인
- [ ] `cpp_doom/build.bat`을 통해 빌드 오류가 없는가?
- [ ] 새로운 기능이 기존 기능(특히 보안/삭제 로직)과 충돌하지 않는가?
- [ ] `_mta.json` 파일의 구조를 변경했다면, 하위 호환성을 고려했는가?
