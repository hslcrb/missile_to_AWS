## 1. aws-nuke의 실제 동작 방식

현재 프로젝트는 `aws-nuke.exe`를 [ProcessHollow](file:///d:/missile_to_AWS/cpp_doom/main.cpp#767-848) 기법으로 메모리에 주입하여 아래 인자를 넘겨 실행합니다:

```
aws-nuke --config external/config.yaml --force
         --access-key-id "..." --secret-access-key "..."
```

`aws-nuke`는 내부적으로 **AWS REST API (HTTPS)**를 직접 호출하여 다음을 수행합니다:

1.  **리소스 열거 (List)**: 각 서비스 API의 `Describe*`, `List*` 엔드포인트 호출
2.  **필터링 (Filter)**: config.yaml의 규칙으로 화이트리스트/블랙리스트 적용
3.  **삭제 (Delete/Terminate/Disable)**: 각 리소스 타입에 맞는 삭제 API 호출
4.  **반복 (Retry)**: 의존성 오류 시 다음 패스에서 재시도

## 2. C++ 순수 구현 시 필요한 핵심 기술 요소

### ① AWS Signature Version 4 (SigV4) 인증
모든 AWS API 호출은 **SigV4 알고리즘**으로 서명되어야 합니다. 이는 다음을 포함합니다:
- SHA-256 HMAC 기반 서명 체인 4단계
- 날짜, 리전, 서비스명, 정규화된 요청 헤더 포함
- C++에서 구현 가능하나 **꽤 복잡한 정밀 구현이 필요**합니다.
- ✅ 현재 프로젝트는 이미 `BCrypt API`를 사용 중이므로 SHA-256은 기반이 있음.

### ② HTTPS 통신 (WinHTTP vs libcurl)
- **WinHTTP.dll**: Windows 기본 제공, 외부 의존성 없음. 단, API가 다소 저수준.
- **libcurl**: 더 편리하나 외부 DLL 필요 — 프로젝트 철학(외부 의존성 최소화)에 반함.
- ✅ `WinHTTP`를 사용해 구현 가능.

### ③ JSON 파싱 (AWS API 응답 처리)
AWS API는 100% JSON 응답을 반환합니다. C++ 표준 라이브러리에는 JSON 파서가 없으므로:
- **직접 구현**: 현재처럼 `wstring::find()` 파싱 — 매우 제한적이고 취약.
- **헤더 전용 라이브러리**: `nlohmann/json.hpp` (단일 헤더 — 의존성 최소) 권장.
- ⚠️ JSON 파싱 없이는 AWS API 응답 처리가 사실상 불가능.

### ④ 400+ 리소스 타입 × N개 리전 지원
`aws-nuke`가 지원하는 AWS 리소스는 **400여 종**입니다. 각 리소스는:
- 별도의 List/Delete API 엔드포인트
- 별도의 파라미터 구조
- 별도의 삭제 조건 (비활성화 후 삭제 등)
- 별도의 페이지네이션 로직 (`NextToken`, `Marker` 등)

이것을 C++로 직접 구현하면 대략 **3,000~10,000줄의 리소스 핸들러 코드**가 필요합니다.

## 4. 핵심 위험 요소 (Risks)

### 🔴 치명적 위험
- **AWS API 스펙 변경**: AWS는 지속적으로 새 서비스/API를 추가합니다. `aws-nuke`는 오픈소스 커뮤니티가 400+ 기여자와 함께 이 사실을 유지합니다. 혼자서는 따라가기 매우 어렵습니다.
- **안전성**: 잘못된 Delete API 호출은 실수로 중요 데이터를 삭제할 수 있습니다. `aws-nuke`는 이 부분에 **10년간의 실전 검증**이 쌓여있습니다.

### 🟡 중간 위험
- **VPC/IAM 의존성 순서**: 리소스 간 삭제 순서가 잘못되면 영구적으로 삭제 불가능한 상태가 됩니다.
- **조건부 삭제**: IAM Role이 실제로 사용 중인지, S3에 잔여 객체가 있는지 등을 모두 사전 체크해야 합니다.

## 5. 대안: 점진적 하이브리드 접근법 (권장)

완전 대체 대신, **선택적 리소스만 직접 C++ API로 처리**하는 방식을 추천합니다:

```
[AWS Cleaner C++]
    ├─ 즐겨찾기 리소스 (EC2, S3, IAM 등) → 직접 WinHTTP API 호출
    └─ 나머지 400+ 리소스 → 기존 aws-nuke 프로세스 주입 유지
```

**이 접근법의 장점:**
- [ProcessHollow](file:///d:/missile_to_AWS/cpp_doom/main.cpp#767-848) 방식을 완전히 버릴 필요 없이, 단계적으로 전환 가능.
- 자주 사용하는 리소스에 한해 진행률 파싱, 확인 메시지 등 UX를 완전 제어 가능.
- SigV4, WinHTTP, JSON 파싱 모듈을 먼저 구현하고 점진적으로 확장.
